01Commands

One Go binary, eight subcommands. Each is a verb the operator runs explicitly — nothing happens implicitly in the background.

CommandWhat it doesKey flags
probe Fetch a live URL, crawl its mindmap, emit a full Playwright + Gherkin suite. The most common starting point for any new site. --url, --coverage (breadth | standard | depth | max), --llm <url>, --ignore-robots, --dry-run
prompt "<text>" Probe a URL but scope to the journey kinds the prompt describes (e.g. "test the checkout flow"). Use --evidence to also run the suite and bundle the report. --url, --coverage, --llm, --evidence
generate Scan a PR's diff, fan source-code symbols out into per-aspect test scaffolds (unit + property + validator + jobs + email + i18n + integration), open a follow-up PR. --pr <N>
scan Dry-run for generate — print every spec it would emit without opening the PR. Use before pushing. --pr <N>
heal Repair broken Playwright locators. On-failure healing reads a Playwright report and proposes resilient selectors; proactive healing scans the whole suite. --pr <N>, --report <file>
ledger update Merge fresh Playwright failures into tests/e2e/docs/findings.md. Re-occurring rows bump LastSeen; resolved rows persist across runs. --report <file>, --out <path>
run-once Run the generated suite locally via npx playwright test --reporter=json. With --record, parses the report and updates the ledger so the next generate emits sentinel specs. --workdir, --record, --report, --grep
Tip. probe generates from a live URL; generate generates from a PR diff. Both can be augmented by the LLM composer via --llm. The composer is strictly opt-in and additive.

02The 10-layer taxonomy

Every test reviewqa emits maps to one of ten layers, modeled after the standard test pyramid expanded for modern web/API surfaces. Layers 1, 2, 6, 9 trigger on PR-diff source code; the other six emit on any live-URL probe.

The "tests per emit" column above reflects the current state.

03Environment variables

Every behavior change can be made via an environment variable or a CLI flag — flags win when both are set. Defaults are tuned for the public-cloud LLM case; local-LLM operators bump REVIEWQA_LLM_TIMEOUT and set REVIEWQA_HUMANIZE=0 for tight wall-clock budgets.

VariableDefaultEffect
REVIEWQA_LLM(empty)OpenAI-compatible endpoint. When set, the composer adds up to 5 @llm-composed Scenarios per journey. Strictly local-only — DGX is on Netbird.
REVIEWQA_MODELgpt-4o-miniModel id used by the composer. Auto-set to qwen3-coder-next:latest when --llm points at the DGX.
REVIEWQA_LLM_LADDER(empty)Comma-separated model fallbacks. First to return parseable JSON wins; the chosen model is embedded as @model:<id> on each emitted Scenario.
REVIEWQA_LLM_TIMEOUT60sPer-call timeout. Bumped from 20s in v0.48 because the DGX takes 20-25s/call. Slower hardware: try 120s+.
REVIEWQA_LLM_TOKEN_CAP600Max output tokens per LLM call.
REVIEWQA_HUMANIZE(unset)Set to 0 to skip the per-file humanization pass while keeping the composer active. Useful when generating hundreds of specs against a slow LLM.
REVIEWQA_ALLOW_DIFF_TO_LLM0Set to 1 to send the PR diff to the LLM for richer scaffold suggestions. Off by default — diff often contains secrets.
REVIEWQA_BROWSER_PROBE(unset)Set to 1 to drive Chromium (via Playwright) instead of the static HTML crawler. Required for SPA targets whose HTML is mostly JavaScript.
REVIEWQA_IGNORE_ROBOTS(unset)Set to 1 to crawl paths a robots.txt Disallow excludes. Default OFF — only enable for QA of sites you own.
REVIEWQA_COVERAGEstandardSpider depth: breadth · standard · depth · max. See the next section.
REVIEWQA_TARGET_URLS(empty)Comma-separated URLs to probe — alternative to --url.
REVIEWQA_HEAL_MODEon-failureon-failure · proactive · off. Controls when the locator-heal command runs.
REVIEWQA_GRAPHQL_ENDPOINT/graphqlOverride the GraphQL stub's introspection path.
REVIEWQA_WEBHOOK_ENDPOINT(empty)Set to the webhook receiver path to activate the signed-POST assertion in pw_webhook_stub.
REVIEWQA_WEBHOOK_SECRET(empty)HMAC signing secret used by pw_webhook_stub.
REVIEWQA_GITHUB_TOKEN(empty)GitHub token used to open the follow-up PR. Falls back to GITHUB_TOKEN.
REVIEWQA_BRANCH_PREFIXreviewqaPrefix for branches reviewqa pushes (e.g. reviewqa/tests-pr-42-abc1234).

04Coverage modes

Four dials, all set in lockstep by one --coverage value. breadth is for CI smoke; standard is the default; depth is for high-stakes audits; max probes wide product surfaces.

ModeMax pagesMax depth (BFS)Journeys per kindFuzz cap
breadth8213
standard (default)30335
depth755610
max1205610
Local-LLM tip. Higher coverage means more journeys, which means more LLM calls. Combine --coverage=depth with REVIEWQA_HUMANIZE=0 so per-file humanization doesn't dominate your wall-clock.

06Workflow recipes

Probe a live URL with the DGX

# Local-only — DGX is reachable via Netbird.
REVIEWQA_LLM=http://100.82.34.115:11434 \
  REVIEWQA_HUMANIZE=0 \
  reviewqa probe --url https://www.spritecloud.com --coverage=standard

Probe an API-only target (lights up the Contract layer)

reviewqa probe --url https://petstore3.swagger.io --coverage=standard
# tests/e2e/contract/ → 12 OpenAPI-derived specs

Generate from a PR diff

reviewqa generate --pr 42
# Scans the diff, emits per-aspect test scaffolds, opens a follow-up PR.

Bootstrap the sentinel layer

# 1) Probe + write the suite to disk
reviewqa probe --url https://example.com

# 2) Install + run it
cd e2e && npm i && npx playwright install

# 3) Run once and record failures into findings.md
reviewqa run-once --record

# 4) Re-probe — each failure now emits a tests/e2e/sentinels/<slug>.sentinel.spec.ts
reviewqa probe --url https://example.com

Run against your own LLM endpoint

REVIEWQA_LLM=https://your-ollama.local:11434 \
  REVIEWQA_MODEL=llama3.1:70b \
  REVIEWQA_LLM_TIMEOUT=120s \
  reviewqa probe --url https://your-site.example.com

07FAQ

My composer keeps failing with unexpected end of JSON input.

Two fixes, both operational: (1) bump REVIEWQA_LLM_TIMEOUT to 120s+ so the model has time to finish its response; (2) set REVIEWQA_HUMANIZE=0 to free LLM budget from the per-file humanization pass. v0.46 also added partial-array recovery — truncated responses contribute whatever Scenarios completed before the cliff.

The probe says "no items produced" against my SPA.

Static HTML crawl can't see JS-rendered content. Set REVIEWQA_BROWSER_PROBE=1 to drive Chromium via Playwright — slower but sees the rendered DOM. Or, for API-only services, point the probe at the OpenAPI root: v0.52 fan-outs the contract + graphql + webhook layers even when no UI pages are discovered.

Where do sentinels come from?

They're emitted from the on-disk ledger at tests/e2e/docs/findings.md. Run reviewqa run-once --record after the first probe to populate it from real test failures; the next generate emits one *.sentinel.spec.ts per finding.

Can I use this in public CI without exposing the LLM?

Yes — never set REVIEWQA_LLM in your CI. The generated .feature and .spec.ts files run on plain Playwright with zero LLM dependency. The composer only runs at generation time, locally.

How do I activate the GraphQL or Webhook stub?

The stub itself always emits per origin. Activate the live assertion by setting REVIEWQA_GRAPHQL_ENDPOINT (default /graphql) or REVIEWQA_WEBHOOK_ENDPOINT + REVIEWQA_WEBHOOK_SECRET. When unset, the test skips with a hint about how to wire it.

What's the difference between probe and generate?

probe takes a live URL and emits the matrix-of-everything against the surface it crawls. generate takes a PR diff and fans the changed source-code symbols out into per-aspect test scaffolds. Both can be augmented by the LLM composer.

How do I customize which taxonomy layers emit?

Most quality companions are unconditional per page — they run cheap and the cost is paid once at generate time. To exclude a layer at runtime, use Playwright's --grep-invert with the tag (e.g. --grep-invert @kind:perf). To exclude a layer at emit time, fork the binary; the conditional gates live in internal/probe/probe.go::qualityCompanions.

How deep is each layer? Are they consistent?

See the "tests per emit" column above. v0.55–v0.57 brought the previously-shallow layers (Contract, Mobile, Integration) up to parity with the deep ones (API, UI, Non-functional). Contract: 9 tests per endpoint including 8 adversarial blocks (sqli / xss / null-byte / oversize / wrong-method / unicode / invalid-json / rapid-burst). Mobile: 8 tests per page across 4 devices × 2 orientations. Integration: 15 tests per origin (5 scaffolds × 3 wireable blocks each, all test.skip() until reviewqa.yml wires the backing resource).

Are the integration stubs always skipped?

Yes, until you wire them. Each per-kind stub (db / cache / obs / auth) ships as test.skip(@wireable, ...) with concrete TODOs in the body — the spec is documentation + catalogue presence. Declare the resource in reviewqa.yml and re-run reviewqa generate to swap the stub for a Testcontainers-backed real spec (integration_db.tmpl, etc.).
