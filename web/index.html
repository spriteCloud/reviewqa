What it does

Deterministic where it matters. Humanized where it helps.

01

Gherkin-first scaffolding

Regex and heuristic extractors map live URLs to user journeys, then emit .feature files in Given/When/Then. Stakeholders read the contract; playwright-bdd runs it.

02

Journeys, priorities, tags

Every test name carries @journey:<kind> and @priority:<level>. Run --grep @priority:critical on every PR; standard nightly; nice-to-have weekly.

03

API contracts + UI flows

Every form on the site gets a sibling *.api.spec.ts — happy, missing-required, malformed-email, oversized, wrong-method — decoupled from the UI happy-flow.

04

Stakeholder docs included

Each run drops tests/e2e/docs/test-catalogue.md (every page, every journey, every priority) and a branded summary.html with a priority-mix bar. Reviewers don't need to read code.

05

Bug-discovery ledger

Failed tests don't die in CI logs — reviewqa ledger update merges them into findings.md, deduped by (spec, test), risk-weighted by priority, surviving across runs.

06

Self‑healing locators

Playwright locator failures get rewritten against the live DOM. Failed CI runs become passing CI runs without manual edits.

07

Quality battery on every probe

Every probe emits accessibility (axe-core), responsive viewports, perf SLO, security headers, health probes, observability headers, visual baselines (toHaveScreenshot), and — when the site advertises them — OpenAPI / GraphQL contract + webhook + i18n specs.

08

gRPC contracts from .proto diffs

Drop a .proto file in your PR and reviewqa emits per-RPC contract tests classified by streaming shape (unary / server-stream / client-stream / bidi) using @grpc/grpc-js.

Coverage knobs

Dial breadth vs. depth without forking the config.

# quick CI smoke (8 pages, 1 journey per kind, 3 fuzz specs)
reviewqa probe --url https://shop.example.com --coverage breadth

# default — current behaviour (20 pages, 3 journeys per kind, 5 fuzz)
reviewqa probe --url https://shop.example.com

# high-stakes audit (50 pages, 6 journeys per kind, 10 fuzz)
reviewqa probe --url https://shop.example.com --coverage depth

# focused, prompt-driven, runs and bundles an evidence ZIP
reviewqa prompt "verify the checkout flow" \
  --url https://shop.example.com --evidence
Drop‑in workflow

One step. Pure Go. No CGO.

Bring any OpenAI‑compatible endpoint — ollama, vLLM, a DGX‑hosted model, or the public API. Leave the key empty and you still get deterministic scaffolds.

# .github/workflows/reviewqa.yml
name: reviewqa
on: pull_request
jobs:
  reviewqa:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: spriteCloud/reviewqa@v1
        with:
          openai-api-key: ${{ secrets.OPENAI_API_KEY }}
          heal-mode: on-failure
Common questions

FAQ

Does reviewqa send my source code to a third-party LLM?

Only when you configure an inference endpoint and your `OPENAI_API_KEY` is set. Even then, the LLM is restricted to rewriting test titles and step comments inside a deterministic scaffold. The PR diff is never sent unless you explicitly opt in with `REVIEWQA_ALLOW_DIFF_TO_LLM=1`.

What happens when the LLM is unavailable or returns garbage?

reviewqa keeps the deterministic scaffold. The LLM rewrite is checked against the original for structural equivalence — same imports, same number of test declarations — and any mismatch is silently discarded.

Can I use my own model instead of OpenAI?

Yes. Any OpenAI-compatible endpoint works — ollama, vLLM, a DGX-hosted model, or a private gateway. Point `OPENAI_BASE_URL` at it and set `REVIEWQA_MODEL` to the model name.
