# v0.61 — spritecloud-com-dgx execution report

The first time the generated spritecloud-com-dgx suite was actually executed
against `https://www.spritecloud.com` via local Chromium. Run on 2026-06-17
against the v0.60-emitted suite. Triaged into three categories per the v0.61
plan:

- **T** — template-level bug (fix in `internal/gen/templates/ts/...`)
- **S** — real spritecloud finding (becomes a sentinel)
- **F** — flaky / LLM-design issue (mitigate in template OR document)

Test environment: Node 20, Playwright 1.61.0, playwright-bdd 9.1.0,
@axe-core/playwright 4.10.0, chromium-1228.

---

## Template bugs found and fixed (T category)

### T-1: playwright-bdd version mismatch in `pw_package.tmpl`

Pre-v0.61 the generated `package.json` pinned `"playwright-bdd": "^7.4.0"`.
`npm install` resolved that to 7.5.0, which `require('playwright/lib/common/configLoader.js')` — an internal API that was removed/moved in playwright 1.51+. Every test run died at config load:

```
Error: Cannot find module '.../node_modules/playwright/lib/common/configLoader.js'
```

**Fix** (`internal/gen/templates/ts/pw_package.tmpl`): bumped to `"playwright-bdd": "^9.0.0"`. Verified `npm install` resolves cleanly to 9.1.0 + the suite loads.

### T-2: `.feature` files prefixed with TypeScript `/* ... */` block comment

`internal/gen/gen.go::annotateQualityReport` prepended a `/* reviewqa
quality report ... */` block to **every** generated file. Gherkin's parser
rejects that with `expected: #EOF, #Language, #TagLine, ..., got '/* reviewqa quality report'`.

**Fix** (`internal/gen/gen.go`): added `plan.TmplPlaywrightFeature` to the
exclusion switch. `.feature` files now use only `#` line comments (which
Gherkin supports natively). The quality-report annotation continues to
attach to all the other `.spec.ts` templates where the JS comment syntax
is valid.

### T-3: `pw_mobile.tmpl` used `test.use()` inside `test.describe()`

The v0.56 mobile template put `test.use({ ...devices[name] })` inside a
`test.describe()` block. Playwright refuses:

> Cannot use({ defaultBrowserType }) in a describe group, because it forces a new worker.

This crashed every mobile spec at collection time.

**Fix** (`internal/gen/templates/ts/pw_mobile.tmpl`): refactored each test
to build the device profile via `browser.newContext({ ...devices[name] })`
explicitly. Same coverage (4 devices × 2 scenarios), no `test.use()`
violation. Verified 32 of 40 mobile tests pass on the live spritecloud
site after the fix.

### T-4: `submit()` helper picked disabled buttons

`pw_steps.tmpl::submit(page)` grabbed `page.getByRole('button', { name: /submit|send|subscribe|.../ }).first()`. On spritecloud the first such button is the footer newsletter "Subscribe" — disabled until you fill its email field. The convert journey was filling the lead form but the submit step would click the newsletter button, which hangs waiting for it to become enabled, then times out after 30s.

**Fix** (`internal/gen/templates/ts/pw_steps.tmpl`): rewrote `submit()` to:

1. Prefer a button inside a `<form>` whose inputs are visible (i.e. the
   form that `fillForm()` likely touched).
2. Scan candidates in order, skip any that's hidden OR disabled.
3. Fall back to a generic visible+enabled `[type=submit]`.
4. Last resort: press Enter (most forms submit on Enter from a filled input).

Verified: convert journey smoke now passes in 2.4s (was timing out at 30s).

## Real spritecloud findings (S category)

These remain as failed tests — they're WCAG violations on the live site,
not problems with reviewqa.

### S-1: 5 footer social-icon links have no accessible name

Affected element class: `.icon_footer.w-inline-block` (LinkedIn, X, YouTube, Medium, brand mark).

Each link is a tab-focusable `<a>` containing only an `<img alt="">` — no `aria-label`, no visible text. Screen-reader users land on focusable elements they cannot identify. axe-core flags this as **`link-name` / serious**.

WCAG: 2.4.4 Link Purpose (Level A), 4.1.2 Name, Role, Value (Level A).

### S-2: `/guides` page has zero `<h1>` tags

Every mobile/landmarks/a11y check that asserts h1 visibility fails on `/guides`. `curl -s https://www.spritecloud.com/guides | grep -c '<h1'` returns 0. Real WCAG 1.3.1 Info and Relationships finding.

### S-3: Color-contrast failures (spot-checked on homepage)

The `@kind:a11y @color-contrast` block flagged violations. Specific elements not enumerated here (full axe-core report is in `tests/e2e/docs/findings.md` after `reviewqa run-once --record`).

### S-4: Unlabeled form inputs

The `@kind:a11y @form-labels` block flagged form inputs without a programmatic label. Confirmed by axe on the same run.

## Flaky / LLM-design issues (F category)

### F-1: LLM-composed scenarios assert against destination-page DOM that doesn't exist

The v0.41a richer prompt gives the composer the destination-page H1s, but composed scenarios sometimes assert against H1 text that isn't on the actual rendered page (e.g. expecting `Case Studies` h1 on a specific case-study page whose actual h1 is the case-study title).

This is the LLM tradeoff — sometimes the model misjudges the contract. Mitigations considered:
- (out of scope for v0.61) Validate composed steps against the destination
  page's actual rendered DOM, not just the registered step pattern.
- Mark `expect.soft()` on the destination-h1 assertion in
  `pw_steps_bdd.tmpl::expectH1()` so the failure surfaces in the report
  without hard-failing the spec. Lower priority because most h1
  assertions on real journey targets pass.

Affected scenarios on this run: 3 of 66 (`@llm-composed @variant @kind:edge` types).
