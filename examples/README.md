# Examples — what `reviewqa probe` produces

Each subdirectory is the complete output of running

```bash
reviewqa probe --url <site> --dry-run
```

against one of the four reference sites in [`docs/reference-sites.md`](../docs/reference-sites.md).
Generated by the v0.30 binary (v1.0-rc); no hand-editing.

v0.24 added diff-mode + OpenAPI-conditional fan-outs (idempotency,
pagination, auth, versioning, schema compatibility, property /
validator / scheduled-job / event-handler / email tests, plus per-store
state tests). None of these light up on the 4 reference sites because
they require either a PR diff or an OpenAPI schema endpoint — but the
templates ship and are exercised by the test suite. See the section
"What v0.24 adds beyond the matrix output" below for what fires when.


## Sites

| Directory | URL | Shape |
|---|---|---|
| [`playwright-dev/`](./playwright-dev/) | <https://playwright.dev> | SaaS-shaped docs, JS-heavy |
| [`gohugo-io/`](./gohugo-io/) | <https://gohugo.io> | OSS marketing + content |
| [`books-toscrape-com/`](./books-toscrape-com/) | <https://books.toscrape.com> | E-commerce list-and-detail |
| [`es-wikipedia-org-madrid/`](./es-wikipedia-org-madrid/) | <https://es.wikipedia.org/wiki/Madrid> | Non-English content page |

## What's in each subdirectory

The same shape every probe run produces — a runnable Playwright + Gherkin
project with stakeholder docs alongside:

```
<site>/
├── playwright.config.ts             # defineBddConfig() + multi-project + cross-browser
├── package.json                     # depends on playwright-bdd + @axe-core/playwright
├── tsconfig.json
├── .github/workflows/e2e.yml
└── tests/e2e/
    ├── features/                    # Gherkin contracts (executable via playwright-bdd)
    │   └── <site>-<journey>.feature
    ├── steps/
    │   └── reviewqa.steps.ts        # Given/When/Then → lib/steps.ts
    ├── lib/
    │   └── steps.ts                 # Steps API helper verbs
    ├── api/                         # API contract specs (when site has <form action>)
    │   └── *.api.spec.ts
    ├── a11y/                        # @axe-core/playwright — no serious/critical violations
    │   └── *.a11y.spec.ts
    ├── responsive/                  # mobile / tablet / desktop viewports
    │   └── *.responsive.spec.ts
    ├── perf/                        # page load < SLO (PERF_SLO_MS env)
    │   └── *.perf.spec.ts
    ├── security/                    # baseline security headers (HSTS, CSP, …)
    │   └── *.security.spec.ts
    ├── health/                      # /health, /healthz, /ready, /status probes
    │   └── *.health.spec.ts
    ├── observability/               # x-request-id, server-timing, traceparent
    │   └── *.observability.spec.ts
    ├── visual/                      # Playwright toHaveScreenshot baselines per viewport
    │   └── *.visual.spec.ts
    ├── contract/                    # OpenAPI / GraphQL contract tests (when schema discovered)
    │   └── *.contract.spec.ts
    ├── webhooks/                    # webhook signature tests (when /webhooks/* detected)
    │   └── *.webhook.spec.ts
    ├── i18n/                        # locale switching (when hreflang siblings present)
    │   └── *.i18n.spec.ts
    ├── *-fuzz.spec.ts               # plain Playwright TS fuzz/negative specs
    ├── _fixtures.ts                 # shared test/expect with page-error tracking
    └── docs/
        ├── test-catalogue.md        # stakeholder doc — every journey + priority
        ├── summary.html             # branded HTML deck with priority-mix bar
        └── findings.md              # bug ledger (seeded empty; persists across runs)
```

## Per-site counts (v0.23)

| Site | Files | .feature | a11y | resp | perf | visual | sec | health | obs | api | fuzz |
|---|---|---|---|---|---|---|---|---|---|---|---|
| `playwright-dev` | 39 | 3 | 5 | 5 | 5 | 5 | 1 | 1 | 1 | 1 | 1 |
| `gohugo-io` | 42 | 3 | 5 | 5 | 5 | 5 | 1 | 1 | 1 | 0 | 5 |
| `books-toscrape-com` | 37 | 3 | 5 | 5 | 5 | 5 | 1 | 1 | 1 | 0 | 0 |
| `es-wikipedia-org-madrid` | 51 | 4 | 5 | 5 | 5 | 5 | 1 | 1 | 1 | 8 | 5 |

Total across the matrix: **169 files**, **13 Gherkin journeys**, **20 each** of a11y / responsive / perf / visual specs, **4 each** of security / health / observability per origin, **9 API contract specs**, **11 fuzz specs**.

Conditional emissions — `contract/` (OpenAPI/GraphQL), `webhooks/`, `i18n/` — fire only when the site advertises a schema, webhook endpoint, or hreflang. None of the four reference sites do, so those directories stay absent here. Pointed at e.g. `https://countries.trevorblades.com` (GraphQL) or `https://petstore.swagger.io` (OpenAPI), they light up. `tests/grpc/` only emits in diff mode when a PR touches `*.proto`.

## What v0.24 adds beyond the matrix output

These spec kinds ship in the v0.24 binary but require triggers that
none of the 4 reference sites provide. The list is what you'll see
the moment the trigger applies:

| Spec kind | Trigger | Output path |
|---|---|---|
| `*.idempotency.spec.ts` | OpenAPI endpoint declares `PUT` or `DELETE` | `tests/e2e/api/<host>-<m>-<slug>.idempotency.spec.ts` |
| `*.pagination.spec.ts` | OpenAPI `GET` on a collection-shaped path (ends in plural noun) | `tests/e2e/api/<host>-<m>-<slug>.pagination.spec.ts` |
| `*.versioning.spec.ts` | OpenAPI advertises both `/v1/...` and `/v2/...` of the same path | `tests/e2e/api/<host>-<m>-<slug>.versioning.spec.ts` |
| `*.compat.test.ts` | PR diff touches `*.proto`, OpenAPI, or AsyncAPI; reviewqa diffs old↔new and surfaces breaking changes | `tests/contract/<schema-slug>.compat.test.ts` |
| `*.property.test.ts` | Diff adds a pure function (no `await` / `fetch` / `console` / `Date.now` / etc) | `tests/property/<stem>.property.test.ts` |
| `*.validator.test.ts` | Function/class name matches `*Validator` / `*Schema` / `validate*` | `tests/validator/<stem>.validator.test.ts` |
| `*.cron.test.ts` | Function uses `cron.schedule` / `@Cron` / `@Scheduled` | `tests/jobs/<stem>.cron.test.ts` |
| `*.event.test.ts` | Function uses `@KafkaListener` / `kafkaConsumer.subscribe` / `@MessageHandler` | `tests/events/<stem>.event.test.ts` |
| `*.email.test.ts` | Function uses `mailer.send` / `sgMail.send` / `nodemailer.*` / `transporter.sendMail` | `tests/email/<stem>.email.test.ts` |

## Re-generating

These outputs are NOT regenerated by CI — they're snapshots committed for
reference. Refresh after a release with:

```bash
go build -o /tmp/reviewqa ./cmd/reviewqa
rm -rf examples/playwright-dev examples/gohugo-io \
       examples/books-toscrape-com examples/es-wikipedia-org-madrid

for spec in \
  "playwright-dev https://playwright.dev" \
  "gohugo-io https://gohugo.io" \
  "books-toscrape-com https://books.toscrape.com" \
  "es-wikipedia-org-madrid https://es.wikipedia.org/wiki/Madrid"; do
    name=$(echo "$spec" | awk '{print $1}')
    url=$(echo "$spec" | awk '{print $2}')
    mkdir -p "examples/$name"
    /tmp/reviewqa probe --url "$url" --dry-run > "/tmp/$name-probe.log" 2>/dev/null
    # split the `--- path ---` blocks into actual files (see explode helper)
done
```

## Why these four sites?

See [`docs/reference-sites.md`](../docs/reference-sites.md) for the rationale +
legal basis per site. They cover a deliberate spread of shapes — JS-heavy
docs, OSS marketing, e-commerce list/detail, non-English content — so the
generator is exercised against more than just one site's vocabulary.
