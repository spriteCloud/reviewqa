# Generated Playwright suite

This directory contains end-to-end tests reviewqa synthesised from
[`https://playwright.dev`](https://playwright.dev). Run them with:

```bash
npx playwright install chromium
npx playwright test
```

## What's here

| File | Purpose |
|---|---|
| `_fixtures.ts` | Shared `test` / `expect` exports with a `pageErrors` fixture. Every spec imports from here. Edit when you want to add suite-wide hooks. |
| `lib/steps.ts` | Higher-level helper verbs (`visit`, `fillForm`, `submit`, `convert`, `authenticate`, …). Compose these in specs instead of repeating click/fill primitives. |
| `*.spec.ts` | One file per user journey reviewqa identified. |
| `docs/test-catalogue.md` | Stakeholder-facing inventory of every page crawled, journey identified, and spec generated — with priority labels. |
| `docs/summary.html` | Branded HTML deck summarising the same coverage. Open in a browser. |
| `../playwright.config.ts` (project root) | Default Chromium config. `BASE_URL` env var overrides the probed origin so the same suite runs against staging, PR previews, or localhost. |

## Priority filtering

Every generated test name carries a `@priority:<level>` tag — `critical`
(convert / contact / authenticate), `standard` (evaluate / research /
browse / discover / exercise), or `nice-to-have` (explore / read).

```bash
npx playwright test --grep @priority:critical   # smoke-of-smokes
npx playwright test --grep @priority:standard   # nightly
npx playwright test --grep @journey:convert     # one kind at a time
```

## Journey kinds

Each spec's filename encodes the user goal it exercises:

- `*-convert.spec.ts` — homepage form / signup. Submits the form.
- `*-contact.spec.ts` — landing → contact page form.
- `*-evaluate.spec.ts` — landing → pricing page.
- `*-research-*.spec.ts` — landing → case-studies list → individual study.
- `*-browse-*.spec.ts` — landing → list page → article / detail.
- `*-discover-*.spec.ts` — landing → service / product page.
- `*-explore-*.spec.ts` — landing → top-nav target.
- `*-read-*.spec.ts` — landing → long-form article.
- `*-exercise.spec.ts` / `*-exercise-<page>.spec.ts` — in-page interactions on that page (search box, accordion, dialog, tab, …).

## Extending

- Add new tests in this directory; they'll inherit the `pageErrors`
  fixture from `_fixtures.ts` automatically as long as you
  `import { test, expect } from './_fixtures'`.
- When reviewqa runs again against this site it AMENDS the
  generated specs in place — your hand-written additions outside the
  `// Generated by reviewqa` blocks survive.
- The reviewqa quality-report header in some specs flags pages where
  no `data-testid` attributes were available. Those are the highest-
  leverage stability improvements you can make.

## Out of scope

The generator deliberately does NOT emit:

- Cross-browser projects (firefox, webkit) — uncomment `projects` in
  `playwright.config.ts` if you want them.
- Mobile viewports — add `test.use({ viewport: ... })` blocks per spec
  you care about.
- Visual regression baselines — `toHaveScreenshot()` requires a
  baseline-management story you'll want to design once.
- Accessibility scans — add `@axe-core/playwright` if you want them.
