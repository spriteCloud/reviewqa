## Journeys identified (3)

Each journey ships as a Gherkin `.feature` file under `tests/e2e/features/`.
playwright-bdd compiles features into Playwright specs at config-load
time via the step definitions in `tests/e2e/steps/`. Journey kinds map
to user goals. Priority groups them for ramp-up runs:

- **critical** — revenue / identity flows (convert, contact, authenticate). Run on every PR.
- **standard** — evaluative flows (evaluate, research, browse, discover, exercise). Run on nightly / pre-merge.
- **nice-to-have** — surface coverage (explore, read). Run on weekly / release.

Filter examples:

    npx playwright test --grep @priority:critical
    npx playwright test --grep @journey:convert
    npx playwright test --grep @smoke

## Assertions emitted

Hard assertions drive the journey: URL changes, h1 visibility, form
submit. Nice-to-have assertions (h2 sub-headings, image alt, meta
description, og:title) are emitted as `expect.soft` so a copy edit
doesn't cascade-fail the journey. Negative tests assert at least one of
"browser blocked submit" / "visible error region" / "no success
message"; sites are free to pick their UX.

## Bug sentinels

Where reviewqa recognises a known-broken pattern (e.g. Webflow
`data-wait` submit), the test is marked `test.fail()` so it stays
visible without burning CI on a known issue. Remove the annotation once
the underlying form handling is sorted.
