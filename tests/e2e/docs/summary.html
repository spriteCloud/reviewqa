At a glance

What's green-lit, what's gated, what's unverified.

quail observed 30 pages on this surface and translated them into 4 user journeys. 2 gate the release — every PR must pass these before merge. 2 run on nightly regression — the early-warning net.

Beyond the journey suite, every crawled page is checked for accessibility (5 axe-core gates, 5 landmark, 5 keyboard), mobile rendering on 4 devices × 2 orientations, network resilience, security headers, and performance budgets. 5 adversarial fuzz specs hammer the discovered forms with 9 negative payloads each.

Unverified surface: the Integration layer (DB / cache / auth / observability) ships as scaffolds that skip until quail.yml binds them to the backing resources. Until then, the suite cannot prove cross-system contracts.

What test layers fired against this surface.

Six layers can be exercised from a live-URL probe. Unit, Component, Backend, and Data require a PR diff and are not in scope for a probe-only run.

UI

exercised
A11y trio (5+5+5 tests) on every crawled page, plus responsive / performance / visual / security / health / observability quality specs.

API

exercised
10 tests per discovered form (happy path + 9 adversarial negatives). When no form was discovered, this layer is dormant.

Contract

always-attempt stubs
OpenAPI / GraphQL / Webhook stubs always probed (6 tests per origin). When the site advertises a schema, full 9-tests-per-endpoint contract specs are emitted.

Integration

scaffold only
5 scaffolds (api / db / cache / observability / auth) × 3 wireable blocks each. Tests skip until quail.yml wires the backing resource.

Mobile

exercised
4 devices (iPhone 13, Pixel 5, iPad Pro 11, Galaxy S9+) × 2 orientations on every crawled page, plus 5 touch-gesture families.

Non-functional

exercised
17+ quality layers including network throttle, race conditions, storage, zoom, print, clipboard, http-chains, file-upload, history-depth, and locale-switch.

Recommended next steps

How to put this suite to work.

1

Install dependencies

Run npm install in this directory to pull @playwright/test and playwright-bdd, then npx playwright install to fetch the browser binaries.

2

Execute the suite

npx bddgen generates the executable specs from the .feature files. Then npx playwright test runs the full suite. Tag-filter with --grep @priority:critical on every PR; nightly the full set.

3

Triage failures

Open tests/e2e/docs/findings.md after each run. quail ledger update --report playwright-report.json deduplicates failures by (spec, test) and surfaces them risk-weighted by priority — survives across runs.

4

Re-probe on change

Re-run quail probe after feature merges to track coverage drift — new pages and new forms get picked up automatically; removed surfaces drop out.

5

Tailor in the browser

quail serve --workdir . opens a local UI where stakeholders can delete journeys, refine Scenarios via chat, suggest locators from the live DOM, and run individual Scenarios — no IDE required.
